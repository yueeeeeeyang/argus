# 开发约束与最佳实践

> 本文件规定了本项目的核心开发约束与代码规范，所有贡献者在编写、修改代码前必须阅读并遵循。

## 1. 需求澄清原则

**在需求不明确时，必须先进行沟通，禁止擅自猜测或简化实现。**

- 若对需求描述存在歧义、遗漏或矛盾，应立即通过 issue、评论或即时通讯工具向需求提出方确认。
- 禁止基于个人假设进行开发，避免因理解偏差导致返工。
- 沟通内容应记录留存，作为后续需求追溯的依据。

## 2. 注释规范

**所有文件、方法、函数、逻辑块必须添加详细且有意义的中文注释。**

- **文件头注释**：每个文件顶部须包含文件职责、创建/修改日期、作者及主要功能概述。
- **方法/函数注释**：须说明其功能、参数含义、返回值及可能抛出的异常；复杂算法需补充思路说明。
- **逻辑块注释**：对非显而易见的业务逻辑、边界处理、临时方案（TODO/HACK）必须 inline 说明原因。
- **禁止**：无意义的注释（如 `// 初始化` 对 `i = 0`）、与代码不符的过时注释、以及直接翻译代码的注释。

## 3. 代码结构与扩展性

**代码结构必须遵循行业最佳实践，并优先考虑后续扩展性。**

- **单一职责**：模块、类、函数应职责单一，避免 God Class / God Method。
- **依赖倒置**：优先依赖于抽象（接口/抽象类），而非具体实现。
- **开闭原则**：对扩展开放，对修改关闭。新增功能应尽量通过新增代码而非修改已有代码来实现。
- **设计模式**：在合适场景下使用成熟的设计模式（如策略模式、工厂模式、观察者模式等），避免过度设计。
- **目录结构**：按功能或领域进行分层（如 `controller/`、`service/`、`repository/`、`domain/`），保持清晰边界。
- **配置外置**：环境相关、可能变更的参数应抽离到配置文件，禁止硬编码。

## 4. 命名规范

**文件名称、接口名称必须清晰、简单，做到"见名知意"。**

- **文件命名**：使用有意义的词汇，反映文件职责；统一使用小写+下划线（`snake_case`）或小驼峰（`camelCase`），并在项目内保持一致。
- **接口/类命名**：使用名词或名词短语，如 `UserRepository`、`OrderService`；避免无意义的缩写（如 `Mgr`、`Util` 除非行业通用）。
- **方法命名**：使用动词或动词短语，明确表达行为，如 `calculateTotalPrice()`、`sendNotification()`。
- **变量命名**：语义明确，避免单字母变量（循环计数器除外）；布尔型变量使用肯定语气，如 `isActive`、`hasPermission`。
- **常量命名**：全大写+下划线，如 `MAX_RETRY_COUNT`。

---

> **注意**：违反以上约束的代码提交将要求修改。请在编码前反复阅读，将规范内化为习惯。
