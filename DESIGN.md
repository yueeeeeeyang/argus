# Argus（阿尔戈斯）设计文档

> **项目名称**：Argus（阿尔戈斯）  
> **项目定位**：高性能日志查看平台（桌面客户端）  
> **技术选型**：Rust + GPUI  
> **核心目标**：性能优先、界面专业简洁、高信息密度

---

## 一、项目概述

### 1.1 项目背景

Argus 是一款面向开发者和运维人员的高性能日志查看平台。在排查系统问题时，开发者经常需要面对动辄数 GB 甚至数十 GB 的日志文件，以及分散在多个目录、压缩包中的日志。传统的日志查看工具（如文本编辑器、命令行工具）在处理超大文件时存在内存占用高、加载缓慢、交互卡顿等问题。

Argus 旨在通过 Rust 的内存安全和高性能特性，结合 GPUI 的 GPU 加速渲染能力，提供一款能够流畅处理超大日志文件的专业工具；对压缩包、远程来源等非随机访问场景按能力分级降级。

### 1.2 核心设计原则

| 原则 | 说明 |
|------|------|
| **性能优先** | 所有设计决策以性能为第一考量，内存占用、渲染帧率、文件加载速度是关键指标 |
| **组件复用** | UI 组件高度抽象，通用组件在不同模块间无缝复用，避免重复造轮子 |
| **专业简洁** | 界面设计专业、克制，去除无关装饰，最大化有效信息展示面积 |
| **高信息密度** | 在有限屏幕空间内展示尽可能多的有效信息，减少用户滚动和切换 |

## 二、技术选型

### 2.1 技术栈总览

| 层级 | 技术选型 | 选型理由 |
|------|----------|----------|
| **编程语言** | Rust | 内存安全、零成本抽象、高性能，适合处理大文件和底层系统操作 |
| **UI 框架** | GPUI | GPU 加速渲染，适合构建高信息密度桌面界面；平台能力以 GPUI 当前稳定支持范围为准 |
| **压缩解压** | `zip` + `tar` + `sevenz-rust` + `bzip2` + `flate2` + `xz2` | MVP 覆盖 ZIP、TAR、tar.gz、tar.bz2、tar.xz、7Z；优先纯 Rust 依赖，RAR 作为可选后续能力 |
| **异步运行时** | `tokio` + `tokio-util` | 行业标准的异步运行时，高效的 I/O 调度、任务取消和后台索引管理 |
| **内存映射** | `memmap2` | 大文件零拷贝读取，避免一次性加载整个文件到内存 |
| **编码检测** | `chardetng` + BOM/UTF-8 校验 | 对未知文本进行启发式检测，检测失败时允许用户手动选择 |
| **文本解码** | `encoding_rs` | 执行具体编码到 UTF-8 的转换，支持增量解码以处理跨页多字节字符 |
| **缓存与凭据** | `lru` + `secrecy` | 控制页缓存上限，并避免密码明文散落在状态、日志和配置中 |

### 2.2 GPUI 架构适配

Argus 将围绕 GPUI 的 `Render`、`Entity` 与 `Context` 能力组织 UI 和状态，遵循以下适配策略：

- **View 层**：基于 `Render` 能力构建声明式 UI，每个视图持有自身状态
- **状态管理**：利用 GPUI 的 `Entity` 和 `Context` 管理视图状态，跨模块共享状态统一进入应用级状态对象
- **增量渲染**：仅对变更区域进行重绘，减少 GPU 负载
- **事件处理**：通过 `subscribe`/`emit` 机制实现组件间解耦通信

### 2.3 目标平台与边界

| 项目 | 策略 |
|------|------|
| **MVP 目标平台** | macOS 优先，Linux 作为第二目标平台验证 |
| **Windows 支持** | 暂不作为 MVP 承诺；若 GPUI 与依赖链路成熟后再评估 |
| **平台相关能力** | 系统主题、文件监听、无默认标题栏窗口、自定义标题栏、快捷键按平台封装，禁止散落在业务层 |
| **窗口装饰策略** | 去除系统默认标题栏，应用内自绘 Obsidian 风格标题栏；macOS 保留交通灯按钮区域，Linux 使用自绘窗口控制按钮 |
| **降级策略** | 平台 API 不可用时保留核心日志查看能力，并在设置页显示能力缺失原因 |

## 四、核心模块设计

### 4.1 日志加载模块（LogFileLoader）

#### 4.1.1 设计目标

- **统一抽象**：文件、目录、压缩包统一抽象为"日志源"，对外暴露一致的访问接口
- **能力分级**：区分普通文件随机访问、压缩条目顺序读取、临时缓存后随机访问三类能力
- **可控落地**：默认优先流式读取；当用户需要跳行、全文搜索或 mmap 时，可按配置把压缩条目缓存到受控临时目录
- **嵌套压缩包支持**：MVP 支持最多 2 层嵌套，后续可通过配置扩展到 5 层
- **密码保护**：支持 ZIP、7Z 密码输入；RAR 密码作为后续可选能力，不纳入 MVP 承诺
- **统一目录树**：构建统一的虚拟目录树，为日志内容读取提供一致入口

#### 4.1.2 核心数据结构

日志加载模块只暴露稳定的来源 ID 和目录树节点，避免 UI 直接持有复杂递归对象。

| 设计对象 | 设计职责 | 关键约束 |
|----------|----------|----------|
| 来源 ID | 作为日志源、目录树节点、读取器、搜索结果之间的关联键 | 全局唯一、轻量、可复制 |
| 来源定位 | 区分本地文件、本地目录、压缩包条目三类来源 | 真实路径和虚拟路径必须分离，禁止用单个字符串混合表达 |
| 来源元信息 | 保存显示名、大小、压缩格式、是否允许临时缓存等信息 | 不保存密码，不保存递归父节点 |
| 目录树节点 | 展示文件、目录、压缩包和压缩包内条目 | 节点仅引用来源 ID，子节点懒加载 |
| 压缩格式 | 表示 ZIP、7Z、TAR 系列、RAR 实验能力和第三方扩展格式 | RAR 默认不进入 MVP 支持范围 |

#### 4.1.3 凭据与来源注册

密码不写入日志源对象、目录树、日志文件或持久化配置，统一交给临时凭据仓库管理：

| 组件 | 职责 | 安全要求 |
|------|------|----------|
| 来源注册表 | 统一维护来源 ID、真实路径、虚拟路径和来源元信息 | UI 与业务模块通过来源 ID 交互 |
| 临时凭据仓库 | 保存压缩包密码等敏感信息 | 仅驻留内存，关闭来源或退出应用时清理 |
| 凭据引用 | 将来源与临时凭据关联 | 不允许把明文密码写入日志、配置或错误信息 |
| 清理流程 | 关闭来源、取消加载、应用退出时触发 | 必须同时释放凭据、临时缓存和未完成任务 |

#### 4.1.4 压缩包适配器架构

采用**适配器模式**封装不同压缩格式。适配器不直接返回依赖局部归档对象生命周期的读取器，而是先创建拥有底层状态的压缩包句柄，再由句柄创建条目读取器：

| 抽象层 | 设计职责 | 关键约束 |
|--------|----------|----------|
| 格式检测器 | 根据文件头、扩展名和内容样本识别压缩格式 | 以文件头为准，扩展名只作为辅助 |
| 压缩包适配器 | 封装 ZIP、7Z、TAR 等格式差异 | 对外暴露统一的条目枚举、条目读取、能力声明 |
| 压缩包句柄 | 持有底层文件和归档状态 | 读取器生命周期不得依赖已释放的局部对象 |
| 条目读取器 | 提供压缩包内单个文件的顺序读取能力 | 默认只保证顺序读取，随机访问需通过临时缓存获得 |
| 能力声明 | 标识来源是否支持随机访问 | 决定后续使用 mmap、顺序预览或临时缓存 |

#### 4.1.5 支持的压缩格式与依赖

| 格式 | 扩展名 | Rust 依赖库 | MVP | 密码支持 | 读取能力 |
|------|--------|-------------|-----|----------|----------|
| ZIP | `.zip` | `zip` | 是 | ZipCrypto/AES 依赖库能力 | 条目流式读取；需要跳行时可临时缓存 |
| 7Z | `.7z` | `sevenz-rust` | 是 | AES-256 依赖库能力 | 通常按条目顺序读取；需要随机访问时临时缓存 |
| TAR | `.tar` | `tar` | 是 | N/A | 顺序读取；普通 tar 条目可按需缓存 |
| GZIP + TAR | `.tar.gz` `.tgz` | `tar` + `flate2` | 是 | N/A | 顺序读取 |
| BZIP2 + TAR | `.tar.bz2` | `tar` + `bzip2` | 是 | N/A | 顺序读取 |
| XZ + TAR | `.tar.xz` | `tar` + `xz2` | 是 | N/A | 顺序读取 |
| RAR | `.rar` | `unrar` 等非纯 Rust/FFI 方案 | 否 | 视后续依赖而定 | 后续评估，默认标记为不支持 |

#### 4.1.6 嵌套压缩包处理流程

```
用户打开 outer.zip
    │
    ▼
[ArchiveDetector] 检测格式为 ZIP
    │
    ▼
[ZipAdapter] 打开 outer.zip，获取条目列表
    │
    ├─ file1.log ────► 直接添加为 ArchiveEntry 节点
    │
    ├─ folder/ ──────► 添加为 ArchiveDirectory 节点，递归展开
    │       │
    │       └─ file2.log ────► 添加为 ArchiveEntry 节点
    │
    └─ inner.tar.gz ─► 检测到为压缩包，标记 is_archive = true
            │
            ▼
        用户展开 inner.tar.gz 节点时才读取并检测格式
            │
            ▼
        小条目：内存缓冲检测；大条目：按配置临时缓存后检测
            │
            ├─ nested1.log ────► 添加为嵌套 ArchiveEntry 节点
            │
            └─ deep.zip ──────► 超过嵌套层级时提示用户并停止展开
```

**嵌套限制**：
- MVP 默认最多 2 层嵌套，避免目录树展开时触发不可控解压。
- 最高可配置到 5 层，超过限制时保留节点但不继续展开。
- 嵌套压缩包必须懒加载，禁止打开外层压缩包时递归扫描所有子压缩包。

#### 4.1.7 流式读取与临时缓存策略

压缩包条目默认通过条目读取器顺序读取；只有当用户触发跳行、全文搜索、随机滚动到未缓存区域等需要随机访问的能力时，才进入可控临时缓存流程：

| 场景 | 默认策略 | 用户可见行为 |
|------|----------|--------------|
| 打开压缩包内小文件 | 顺序读取并预览 | 立即展示首屏内容 |
| 打开压缩包内大文件 | 顺序读取首屏，需要随机访问时再缓存 | 状态栏提示当前为预览模式 |
| 跳转到指定行 | 若来源不可随机访问，先请求临时缓存授权 | 显示缓存大小估算和取消入口 |
| 全文搜索 | 随机访问来源直接搜索；顺序来源按策略缓存或降级为顺序扫描 | 显示进度、结果上限和取消入口 |
| 嵌套压缩包展开 | 小条目内存检测，大条目临时缓存检测 | 超过层级或缓存上限时给出提示 |

**内存控制策略**：
- 流式读取使用固定大小的缓冲区（默认 64KB），避免内存暴涨
- 临时缓存有单条目和会话总量上限，超限时提示用户选择继续、取消或仅预览
- 缓存文件使用随机文件名并记录来源 ID，关闭来源时立即清理
- 嵌套压缩包展开、全文搜索和跳行都必须可取消，取消后清理未完成缓存

### 4.2 日志内容读取模块（LogFileReader）

#### 4.2.1 设计目标

- **统一读取接口**：通过日志加载模块提供的目录树和文件信息，统一读取任意日志内容
- **大文件友好**：支持从几 KB 到数十 GB 的日志文件，避免全量载入内存
- **后端分级**：普通文件使用 mmap，压缩流使用顺序读取，必要时临时缓存后再 mmap
- **高效行定位**：支持快速跳转到指定行号、搜索结果的精确行定位
- **安全解码**：使用 BOM、UTF-8 校验、`chardetng` 启发式检测和手动选择组合识别编码
- **跨页正确性**：分页边界对齐换行，增量解码器处理跨页多字节字符和跨页长行

#### 4.2.2 核心数据结构

读取模块围绕“来源 ID、读取后端、分页状态、编码决策、页缓存”组织，不把具体文件、mmap 或压缩流细节泄漏给 UI。

| 设计对象 | 设计职责 | 关键约束 |
|----------|----------|----------|
| 读取后端类型 | 标识普通文件 mmap、顺序流、临时缓存后 mmap 三种访问方式 | 后端类型由加载模块按能力选择 |
| 读取模式 | 标识直接模式、分页模式、顺序预览模式 | 文件大小只是因素之一，随机访问能力优先级更高 |
| 索引状态 | 描述行索引未开始、构建中、部分可用、完整可用 | 跳行和搜索必须能处理索引未完成状态 |
| 分页信息 | 保存目标页大小、真实页边界、页起始行号和总行数 | 页边界必须对齐换行，不能按固定字节硬切 |
| 解码页面 | 保存页号、原始字节范围、解码后的行和续行标记 | 搜索高亮必须能回到原始字节范围 |
| 页缓存 | 缓存最近访问的解码页 | 必须有 LRU 上限，避免随滚动无限增长 |

#### 4.2.3 读取后端抽象

读取模块不假设所有来源都能 mmap，而是通过读取后端抽象能力：

| 后端 | 适用来源 | 能力 | 降级策略 |
|------|----------|------|----------|
| 普通文件 mmap 后端 | 本地普通文件、已缓存临时文件 | 支持随机访问、分页、跳行、全文搜索 | mmap 失败时降级为分块读取 |
| 顺序流后端 | 压缩包条目、未来的管道或远程流 | 支持顺序预览和连续读取 | 需要随机访问时请求临时缓存 |
| 临时缓存后 mmap 后端 | 已授权缓存的压缩条目 | 支持与普通文件接近的随机访问体验 | 缓存超限或取消时回到顺序预览 |

后端选择规则：
- 普通本地文件优先使用普通文件 mmap 后端。
- 压缩包内条目默认使用顺序流后端做顺序预览。
- 当用户触发跳行、全文搜索、随机滚动到未缓存区域时，加载模块按临时缓存策略转换为临时缓存后 mmap 后端。
- 用户关闭临时缓存或缓存超限时，UI 显示能力降级提示，不伪装成完整随机访问。

#### 4.2.4 编码检测与增量解码

编码检测分为四步，检测结果必须带来源和置信度：

| 检测来源 | 优先级 | 说明 |
|----------|--------|------|
| BOM | 最高 | 明确识别 UTF-8、UTF-16 等带 BOM 文件 |
| UTF-8 严格校验 | 高 | 无 BOM 但完整样本满足 UTF-8 时直接采用 |
| 启发式检测 | 中 | 使用 `chardetng` 处理 GBK、GB2312、Latin-1 等常见文本 |
| 用户手动选择 | 高于自动结果 | 用户选择后覆盖自动检测结果，并刷新当前视图 |
| 默认编码兜底 | 最低 | 检测置信度不足时使用配置默认值 |

增量解码器必须保留跨页未完成字节，避免 UTF-8、GBK 等多字节字符在页边界处被拆坏。用户切换编码时，已缓存页面必须失效并重新解码。

**检测流程**：
1. 读取前 64KB 样本，优先识别 UTF-8/UTF-16 BOM。
2. 无 BOM 时做 UTF-8 严格校验，通过则使用 UTF-8。
3. UTF-8 校验失败时调用 `chardetng` 启发式检测，并结合用户地区偏好。
4. 置信度不足时使用默认编码并在状态栏提供手动切换入口。

#### 4.2.5 分页与行索引策略

分页目标大小默认 1MB，但真实页边界必须对齐到换行符，避免跨页切断日志行：

| 策略 | 说明 |
|------|------|
| 首屏优先 | 打开文件时只构建首屏必需的页边界，不同步扫描完整文件 |
| 换行对齐 | 每个页面从行首开始，页尾向后扫描到下一个换行符 |
| 超长行处理 | 单行超过阈值时允许拆成多个显示片段，并标记为同一逻辑行续段 |
| 后台索引 | 行号索引按块构建，持续上报已扫描字节数和估算进度 |
| 可取消 | 用户关闭文件、重新打开、切换来源时必须取消索引任务 |
| 部分可用 | 已扫描范围内允许跳行和搜索定位，未扫描范围提示等待或继续构建 |


#### 4.2.6 分页读取与跳行策略

| 操作 | 设计行为 |
|------|----------|
| 读取当前页 | 根据页边界读取原始字节，使用增量解码生成显示行 |
| 快速滚动 | 只加载可见范围及少量缓冲页，旧页面按 LRU 释放 |
| 跳转到行 | 索引已覆盖目标行时立即定位；未覆盖时显示索引进度并允许取消 |
| 搜索结果跳转 | 优先使用搜索结果字节范围定位，再映射到页和行 |
| 编码切换 | 清空解码页缓存，保留原始页边界和行索引，重新解码可见页 |

#### 4.2.7 读取模式选择

| 条件 | 读取模式 | 原因 |
|------|----------|------|
| 随机访问来源且小于直接模式阈值 | 直接模式 | 小文件一次性读取可以简化交互并提升响应 |
| 随机访问来源且大于等于阈值 | 分页模式 | 避免大文件占用过多内存 |
| 顺序来源且用户只预览 | 顺序预览模式 | 不强制落地临时文件 |
| 顺序来源且需要跳行/全文搜索 | 请求临时缓存后分页模式 | 随机访问能力需要可 seek 的后端支持 |
| 缓存被禁用或超限 | 保持顺序预览并提示能力受限 | 不隐式消耗磁盘空间 |

#### 4.2.8 行号计算与缓存

行号索引记录每行起始字节、长度和分页映射，用于跳行、搜索结果定位和状态栏展示。索引构建必须分块执行，并正确处理 LF、CRLF、文件末尾无换行和超长行。索引数据可以在当前会话内缓存，但持久化索引不纳入 MVP。

### 4.3 搜索分析模块（SearchEngine）

#### 4.3.1 设计目标

- **流式搜索**：按页或按块扫描，禁止为搜索一次性加载完整文件。
- **可取消与可观察**：每个搜索任务都有进度、取消标记和状态栏反馈。
- **结果有上限**：默认最多保留 10,000 条结果，避免海量匹配耗尽内存。
- **正则安全**：限制正则复杂度、单块执行时间和最大匹配长度，防止搜索卡死。
- **精确定位**：搜索结果保存字节范围、行号和页号；行号索引未完成时先保存字节范围，索引完成后补齐行号。

#### 4.3.2 核心数据结构

| 设计对象 | 设计职责 | 关键约束 |
|----------|----------|----------|
| 搜索请求 | 保存关键字、搜索模式、大小写、全词匹配和最大结果数 | 实时搜索会取消上一轮未完成请求 |
| 搜索任务 | 跟踪待开始、运行中、已取消、已完成、失败等状态 | 必须可取消并持续上报进度 |
| 搜索结果 | 保存来源 ID、字节范围、行号、页号和短预览 | 行号索引未完成时先保存字节范围 |
| 正则缓存 | 缓存已编译表达式 | 缓存有容量上限，表达式无效时立即反馈 |
| 结果集合 | 保存当前搜索命中 | 默认最多保留 10,000 条，达到上限后提示收窄条件 |

#### 4.3.3 执行策略

| 来源能力 | 搜索策略 |
|----------|----------|
| 随机访问来源 | 按页或按块并行扫描，结果流式返回 |
| 顺序流来源 | 顺序扫描已读取内容；需要完整可跳转结果时请求临时缓存 |
| 编码未确认 | 使用当前编码决策搜索，并允许用户切换编码后重新搜索 |
| 索引未完成 | 先返回字节范围，索引补齐后异步回填行号和页号 |
| 用户继续输入 | 200ms debounce 后启动新任务，并取消上一轮搜索 |

**正则安全策略**：
- 默认使用 Rust `regex` 引擎，避免回溯型正则的灾难性回溯。
- 限制单个表达式长度、最大捕获组数量和最大匹配文本长度。
- 搜索压缩包内顺序流时，如用户要求全文可跳转结果，需先执行临时缓存流程。
- 实时输入搜索使用 200ms debounce，并自动取消上一轮未完成任务。

### 4.4 UI 组件层（Component Layer）

#### 4.4.1 设计原则

- **通用复用**：所有组件与业务逻辑解耦，可在不同场景下复用
- **性能优先**：大数据量场景使用虚拟滚动，仅渲染可见区域
- **高信息密度**：组件布局紧凑，减少内边距和间距，最大化内容展示

#### 4.4.2 核心组件清单

| 组件名称 | 组件用途 | 关键特性 |
|----------|----------|----------|
| `CustomTitleBar` | 自定义标题栏，用于替代系统默认标题栏 | 集成窗口拖拽区、文件标签、返回/前进、全局操作和窗口控制 |

| `SearchBox` | 搜索输入框 | 正则切换、大小写切换、实时高亮 |

#### 4.4.3 自定义标题栏（CustomTitleBar）设计

自定义标题栏参考 Obsidian 的顶部结构，去除系统默认标题栏后由应用统一绘制。标题栏左侧预留平台窗口控制区，中部承载标签页，右侧放置全局操作入口。

| 设计点 | 策略 |
|--------|------|
| 窗口控制 | macOS 保留交通灯按钮安全区域；Linux 使用自绘最小化、最大化、关闭按钮 |
| 拖拽区域 | 标签页外的空白区域可拖拽窗口，双击执行最大化或还原 |
| 标签呈现 | 当前标签使用更高对比背景和圆角连接，非活动标签弱化显示 |
| 全局操作 | 新建标签、搜索、设置、布局切换等放在标题栏两侧，不提供书签入口 |
| 内容面包屑 | 主内容区顶部可显示当前来源路径，避免标题栏承载过多文本 |
| 平台适配 | 所有窗口命令通过平台适配层调用，业务组件不直接处理系统窗口 API |

#### 4.4.4 活动栏（ActivityBar）设计

左侧活动栏为固定宽度纵向图标导航，类似 Obsidian 左侧 ribbon。当前阶段只保留“日志分析”和“设置”两个入口，避免把搜索、书签、历史等后续能力提前暴露到主导航。

| 设计点 | 策略 |
|--------|------|
| 固定位置 | 位于窗口最左侧，贯穿标题栏下方到窗口底部，不受内容区状态栏影响 |
| 固定宽度 | 默认 44-52px，禁止随侧栏展开而变宽 |
| 图标导航 | 只显示图标，悬停显示 Tooltip，选中项使用高亮背景或侧边指示条 |
| 工作区切换 | 点击图标在日志分析工作区和设置工作区之间切换 |
| 底部操作 | 设置入口固定在底部，日志分析入口固定在顶部 |
| 可访问性 | 所有图标必须有可读标签和键盘焦点状态 |

#### 4.4.5 虚拟列表（VirtualList）设计

虚拟列表是日志查看器的核心组件，决定大文件的滚动性能：

| 设计点 | 策略 |
|--------|------|
| 数据来源 | 不持有全量行，只按可见范围向读取器请求数据 |
| 可见范围 | 根据滚动偏移、视口高度、行高和缓冲区计算 |
| 总行数未知 | 使用估算滚动条，索引完成后平滑校正 |
| 动态行高 | 缓存已测量行高，避免重复布局 |
| 快速滚动 | 优先展示占位和最近缓存页，后台补齐真实内容 |

**性能优化要点**：
- 使用缓冲区上下各预渲染少量不可见项，减少快速滚动时的空白闪烁
- 滚动事件按帧节流（throttle），避免高频事件触发重复读取
- 行高缓存避免重复测量，提升动态行高模式性能
- 总行数未知时使用估算滚动条；后台索引完成后平滑校正滚动范围

#### 4.4.6 树形控件（TreeView）设计

树形控件用于展示本地目录、压缩包和压缩包内条目。节点展开时才加载子节点，避免打开大目录或大压缩包时阻塞 UI。

| 设计点 | 策略 |
|--------|------|
| 懒加载 | 目录和压缩包节点展开时才读取子项 |
| 选择状态 | 选中节点只保存节点 ID 和来源 ID |
| 展开状态 | 展开/折叠状态由 UI 层维护，可按来源恢复 |
| 大目录 | 子项加载分批进行，状态栏显示进度并支持取消 |
| 图标与标签 | 根据文件、目录、压缩包、压缩包内条目区分视觉状态 |

### 5.1 主窗口布局

### 5.2 布局设计原则

1. **Obsidian 风格框架**：窗口由自定义标题栏、左侧活动栏、来源侧栏和主内容区组成，状态栏归属于主内容区。
2. **左侧纵向图标导航条**：活动栏固定在最左侧，当前只提供“日志分析”和“设置”两个入口。
3. **去除默认窗口标题栏**：应用使用无默认标题栏窗口，自绘顶部标题栏和标签页区域，形成统一视觉。
4. **标签式标题栏**：当前日志文件以标签形式展示，支持关闭、新建、切换和拖拽排序。
5. **来源侧栏可折叠**：目录树和搜索结果等侧栏内容可折叠，折叠后仅保留活动栏，最大化日志阅读空间。
6. **上下文工具内聚**：打开、刷新、折叠、编码、换行等操作放入标题栏、侧栏工具区或内容区右上角，避免单独占用整行工具栏。
7. **内容区状态栏**：状态栏只显示在主内容区域下方，不覆盖左侧活动栏和来源侧栏；文件编码、行数、大小、光标位置、索引进度、临时缓存状态和加载状态一目了然。
8. **紧凑间距**：UI 元素间距控制在 2-4px，行高 1.4 倍，最大化信息密度。

### 5.3 自定义标题栏设计

| 区域 | 内容 | 设计要求 |
|------|------|----------|
| 左侧窗口控制区 | macOS 交通灯按钮或 Linux 自绘窗口按钮 | 必须预留系统安全区域，不能被标签或图标遮挡 |
| 左侧快捷操作 | 打开来源、全局搜索等常用入口 | 与日志分析工作区相关，去除书签入口 |
| 中部标签区 | 当前打开的日志文件、关闭按钮、新建标签按钮 | 当前标签高亮，非活动标签弱化，长文件名省略显示 |
| 空白拖拽区 | 标签外的顶部空白区域 | 支持拖拽窗口和双击最大化/还原 |
| 右侧全局操作 | 布局切换、设置、更多菜单 | 图标化展示，悬停显示说明 |

## 六、数据流设计

### 6.1 文件加载流程

```
用户操作（打开文件/目录）
    │
    ▼
[UI 层] 调用 LogFileLoader::open()
    │
    ▼
[LogFileLoader]
    ├─ 检测路径类型（文件/目录/压缩包）
    ├─ 若为压缩包：调用 ArchiveDetector 识别格式
    │   └─ 调用对应 ArchiveAdapter 打开
    │       └─ 只枚举当前层条目，嵌套压缩包懒加载
    ├─ 将来源写入 SourceRegistry
    └─ 构建统一 DirTree（节点仅保存 SourceId）
    │
    ▼
返回 DirTree 给 UI 层
    │
    ▼
[UI 层] 在来源工作区渲染来源侧栏
    │
    ▼
用户点击文件节点
    │
    ▼
[UI 层] 调用 LogFileReader::open()
    │
    ▼
[LogFileReader]
    ├─ 根据来源能力选择 PlainMmap / SequentialStream / SpoolThenMmap
    ├─ 执行 BOM、UTF-8 校验、chardetng 检测或用户编码覆盖
    ├─ 构建首屏页边界，真实页边界对齐换行
    ├─ 后台分段构建行索引
    └─ 返回读取器实例
    │
    ▼
[UI 层] 通过 VirtualList 渲染日志内容
    │
    ▼
用户滚动/搜索/跳转
    │
    ▼
[LogFileReader/SearchEngine] 按需读取对应页/行，必要时触发临时缓存或等待索引
```

### 6.2 状态管理

采用 GPUI 的 **Entity + Context** 模式进行状态管理：

| 状态对象 | 职责 | 约束 |
|----------|------|------|
| 来源注册表 | 维护所有打开来源、真实路径、虚拟路径和来源元信息 | UI 只保存来源 ID，不复制来源结构 |
| 临时凭据仓库 | 保存压缩包密码等敏感数据 | 仅驻留内存，关闭来源时清理 |
| 当前目录树 | 表示当前打开来源的可视化结构 | 节点懒加载，节点只引用来源 ID |
| 读取器集合 | 按来源 ID 管理已打开读取器 | 关闭来源时同步释放读取器、页缓存和后台任务 |
| 搜索状态 | 记录当前搜索任务、进度、结果数量和错误 | 新搜索开始时取消旧任务 |
| 主题管理器 | 管理当前主题、系统主题事件和订阅者通知 | 平台事件通过主循环处理 |
| 应用配置 | 保存编码、分页、临时缓存、搜索上限、字体、主题等用户设置 | 环境相关参数必须外置到配置文件 |

状态流转遵循“UI 事件 -> 应用状态更新 -> 组件重新渲染”的单向路径。耗时任务只通过事件或消息回写状态，禁止直接在后台线程修改 UI 状态。

## 七、性能优化策略

### 7.1 文件加载性能

| 优化点 | 策略 | 效果 |
|--------|------|------|
| **后端分级** | 普通文件 mmap，压缩条目顺序读，必要时临时缓存后 mmap | 避免把不可随机访问的来源误用为 mmap |
| **流式解压** | 压缩包默认惰性读取，跳行/全文搜索时按策略临时缓存 | 控制磁盘 I/O，同时保留大文件随机访问能力 |
| **异步索引** | 首屏只构建必要页边界，行号索引后台分段构建 | 文件打开后可立即查看，无需等待完整索引 |
| **懒加载目录** | 目录内容按需展开加载 | 避免一次性读取成千上万个文件信息 |
| **编码预检测** | 使用 BOM、UTF-8 校验和 chardetng 样本检测 | 避免全文扫描，并允许用户手动覆盖 |

### 7.2 渲染性能

| 优化点 | 策略 | 效果 |
|--------|------|------|
| **虚拟列表** | UI 仅按可见范围请求数据，不持有全量行 Vec | GPU 和内存负载随视口大小保持稳定 |
| **增量渲染** | GPUI 自动仅重绘变更区域 | 减少 GPU 绘制调用 |
| **文本布局缓存** | 缓存可见窗口附近的文本布局结果 | 降低重复布局成本，避免无边界 GPU 纹理缓存 |
| **异步搜索** | 搜索任务可取消、限结果数、流式返回进度 | UI 不卡顿，支持大文件搜索 |
| **分页缓存** | LRU 缓存最近访问的页 | 频繁切换页无需重新读取 |

### 7.3 内存优化

| 优化点 | 策略 | 效果 |
|--------|------|------|
| **能力驱动模式** | 先看后端是否随机访问，再结合 50MB 阈值选择 Direct/Paged | 避免压缩流被错误当成普通文件处理 |
| **页缓存上限** | LRU 缓存最多 10 页（可配置） | 防止大文件浏览时内存持续增长 |
| **临时缓存上限** | 单条目和单会话缓存均可配置并可取消 | 防止压缩包条目解压耗尽磁盘 |
| **流式释放** | 关闭文件后立即释放所有相关内存 | 避免内存泄漏 |
| **预览文本裁剪** | 搜索预览和状态栏文本限制长度 | 避免长行复制导致额外内存峰值 |

## 八、扩展性设计

### 8.1 压缩格式扩展

新增压缩格式需要同时提供格式识别、条目枚举、条目读取、能力声明、密码处理和错误映射。扩展格式默认先以顺序读取能力接入；只有明确支持随机访问或完成临时缓存适配后，才允许启用跳行和全文搜索。

| 扩展点 | 要求 |
|--------|------|
| 格式识别 | 提供文件头识别规则，扩展名只作为辅助 |
| 条目枚举 | 支持大压缩包分批枚举，避免一次性加载全部条目 |
| 条目读取 | 明确声明顺序读取或随机访问能力 |
| 密码处理 | 使用临时凭据仓库，不持久化明文密码 |
| 错误映射 | 将格式错误、密码错误、损坏文件、能力不支持映射为用户可理解错误 |


### 8.2 主题扩展

主题通过 JSON/TOML 文件定义，放置在 `~/.config/argus/themes/` 目录即可自动加载：

主题文件只允许声明令牌值，不允许包含脚本或动态逻辑。加载主题时需要校验颜色格式、必填令牌和明暗模式声明；缺失令牌使用内置主题兜底，并在设置页提示。

### 8.3 编码扩展

编码能力分为检测和解码两层。`encoding_rs` 负责解码；未知编码检测由 BOM、UTF-8 校验、`chardetng` 和用户手动选择共同完成。如需支持特殊编码，需要提供编码标签识别、批量解码、增量解码和错误恢复策略。

| 扩展点 | 要求 |
|--------|------|
| 标签识别 | 支持标准编码标签和常见别名 |
| 批量解码 | 用于小文件和搜索预览 |
| 增量解码 | 用于分页读取，必须处理跨页多字节字符 |
| 错误恢复 | 解码失败时允许替换字符或提示用户切换编码 |


### 8.4 日志解析扩展

支持插件化日志解析器，自动识别日志格式并高亮：

日志解析器以“检测 -> 解析 -> 高亮 -> 结构化字段”四步接入。解析器必须逐行工作，不能要求一次性读取完整文件。解析失败时应回退到纯文本展示，不影响基础日志查看能力。

| 扩展点 | 要求 |
|--------|------|
| 格式检测 | 基于首批样本行判断是否适用 |
| 行解析 | 输出时间戳、级别、模块、消息和附加字段 |
| 高亮规则 | 只作用于可见行，避免全文件预处理 |
| 失败回退 | 单行解析失败不影响其他行显示 |

## 十、错误处理策略

### 10.1 错误类型设计

错误类型按模块归属拆分，并在应用层统一转换为用户可理解提示。

| 错误类别 | 典型场景 | 用户提示策略 |
|----------|----------|--------------|
| 加载错误 | 文件不存在、权限不足、目录过大、格式不支持 | 展示路径和重试/重新选择入口 |
| 读取错误 | 随机访问不可用、页范围越界、文件被外部修改 | 展示当前能力限制，并提供刷新或缓存选项 |
| 压缩包错误 | 密码错误、文件损坏、嵌套过深、格式暂不支持 | 提供重新输入密码、停止展开或查看支持格式 |
| 编码错误 | 检测置信度不足、解码失败、用户编码不匹配 | 提供编码选择列表并允许重新解码 |
| 临时缓存错误 | 空间不足、超出上限、用户取消、缓存清理失败 | 展示所需空间、上限和取消/调整入口 |
| 搜索错误 | 正则无效、结果过多、任务取消、来源不可搜索 | 在搜索区域就地提示，不阻断日志浏览 |
| 配置错误 | 配置文件损坏、主题文件缺失、无效配置值 | 使用默认配置启动，并提示修复配置 |

### 10.2 用户友好错误提示

- **文件不存在**：弹窗提示文件路径，提供"重新选择"按钮
- **压缩包密码错误**：弹出密码输入框，支持重新输入
- **编码识别失败**：提供编码手动选择列表
- **大文件加载慢**：状态栏显示加载进度，支持取消操作
- **嵌套压缩包过深**：友好提示"嵌套层级超过限制"
- **随机访问不可用**：提示压缩包条目需要临时缓存才能跳行或全文搜索，并提供"允许缓存"按钮
- **临时缓存超限**：显示所需空间和当前配置上限，允许用户调整上限、仅预览或取消
- **正则表达式无效/过大**：在搜索框下方展示错误原因，不启动搜索任务
- **搜索结果过多**：达到结果上限时停止收集并提示用户收窄条件
